# Pocketo — Roadmap

This is a working backlog for anyone picking up the project. Each item says **what**, **why**,
**where in the code** to start, a rough **approach**, an **effort** estimate (S ≈ a few hours,
M ≈ a day or two, L ≈ a week+), and **acceptance criteria** so "done" is unambiguous.

Read [`PROJECT_GUIDE.md`](PROJECT_GUIDE.md) first — it explains the codebase file by file.
Keep the project's two rules in mind while building:

1. **Correctness-critical logic is a pure function in `src/core/` with a `.test.ts` beside it.**
   Never put money/tax/date math in a component.
2. **Don't claim accessibility you didn't measure.** If you touch colours or markup, re-run
   `npm run test:e2e` (it includes the axe audit).

Status legend: 🟢 ready to start · 🟡 needs a small design decision first · 🔴 needs an
external account or a product/owner decision before any code.

## Already shipped (v0.1)

Income/expense logging, pockets with auto-allocation, kakeibo reports, per-category budgets,
recurring (weekly/monthly/yearly with catch-up), Thai tax estimator (40(1)), searchable
history with edit/undo, encrypted JSON backup, onboarding, dark/light/system themes (WCAG AA),
installable offline PWA, OG preview, CI + auto-deploy. See [`CHANGELOG.md`](CHANGELOG.md).

## Near-term — highest value per effort

### 1. 🟢 Scope `History` and `Home` queries (performance) — S
- **Why:** Every screen except Reports still loads the whole `tx` table into memory via
  `db.tx.toArray()`. Fine for hundreds of rows; sluggish at ~10k+ on old phones.
- **Where:** `src/screens/History.tsx`, `src/screens/Home.tsx`. Reports already shows the
  pattern (`src/screens/Reports.tsx` queries a date range through the `date` index).
- **Approach:** In History, when there's no search query, page through `db.tx.orderBy("date")
  .reverse()` with `.offset()/.limit()` instead of loading all then slicing. Keep the
  full-scan path only when a search term is present. Home needs all-time data for balances,
  but the "recent" list and month totals can use a bounded range; balances can stay as-is or
  move behind a memoized cache.
- **Acceptance:** Same UI behaviour; no `toArray()` of the full table on first paint of
  History; add a test seeding 2k transactions and asserting History renders.

### 2. 🟡 Recurring rules: end date + "every N" interval — M
- **Why:** Real subscriptions end; some bills are every 2 weeks or every 3 months.
- **Where:** `src/core/recurring.ts` (pure engine + tests), `src/core/types.ts` (`Recurring`),
  `src/components/RecurringManager.tsx` (UI), bump schema in `src/db/db.ts` if new indexed
  fields, extend `src/core/backup.ts`.
- **Approach:** Add optional `until?: string` and `interval?: number` to `Recurring`. Extend
  `dueDates`/`nextOccurrence` to step by `interval` and stop at `until`. Add unit tests for the
  new cases first.
- **Acceptance:** New fields round-trip through backup (schemaVersion bump, old files still
  import); engine tests cover end-date and N-interval; UI lets you set them.

### 3. 🟢 Local reminders/notifications — M
- **Why:** "upcoming" recurring items and budget overruns are more useful as a nudge.
- **Where:** new `src/core/notify.ts` (permission + scheduling helpers), wire from
  `src/App.tsx`/`src/screens/Home.tsx`; the service worker is generated by `vite-plugin-pwa`
  (`vite.config.ts`).
- **Approach:** Use the Notification API for in-session nudges first (no server needed). True
  background push needs a push service and is out of local-first scope — keep it to
  on-open reminders ("3 recurring items due", "Food budget 90% used").
- **Acceptance:** Opt-in toggle in Settings; nothing fires without permission; works offline.

## Mid-term — meaningful features

### 4. 🔴 Opt-in cross-device sync — L  *(the retention unlock)*
- **Why:** Today, moving devices means hand-carrying a JSON file. This is the single biggest
  ceiling on the product (see the CEO/CTO review).
- **Needs first:** a backend decision and account — recommended **Supabase free tier**
  (Postgres + Auth). This is the 🔴: don't start coding until the owner picks the provider.
- **Where:** keep the local-first core untouched. Add a thin `src/sync/` layer that pushes/pulls
  the same shapes `src/core/backup.ts` already produces. Auth (Google) only for users who opt in;
  everyone else keeps working fully local.
- **Approach:** Per-row `updatedAt` + soft-delete tombstones for last-writer-wins; sync the four
  tables; never block the UI on the network. Encrypt-at-rest if feasible to preserve the privacy
  promise (reuse `src/core/crypto.ts` ideas).
- **Acceptance:** A user with no account sees zero change. A signed-in user editing on two
  devices converges without losing data; offline edits sync on reconnect; documented in
  `PRIVACY.md`.

### 5. 🟢 Bank-statement CSV import — M
- **Why:** Power users don't want to hand-type a month of transactions.
- **Where:** new `src/core/csv.ts` (pure parser + tests), an import wizard component, write via
  the existing `saveQuickTx`/bulk path in `src/db/data.ts`.
- **Approach:** Parse with a tiny CSV reader (no heavy dep), let the user map columns
  (date/amount/description) and a default category/pocket, preview, then bulk insert. Dedupe by
  (date, amount, description) against existing rows.
- **Acceptance:** Parser is unit-tested with messy real-world CSVs; preview before commit; no
  duplicate rows on re-import of the same file.

### 6. 🟡 Tax engine phase 2 — M
- **Why:** Most filers have family allowances and sometimes non-salary income.
- **Where:** `src/core/tax.ts` + `src/core/tax.test.ts`, then surface fields in
  `src/screens/Tax.tsx`.
- **Approach:** Add allowances to the config/`Deductions`: spouse (60k), children (30k, 60k for
  the 2nd+ born ≥2561), parents (30k each), Easy-E-Receipt, antenatal, etc. Then support income
  types beyond 40(1) (e.g. 40(2)) with their own expense-deduction rules. **Add tests for every
  new cap before wiring UI.** Keep rules in the per-year config object, not in logic.
- **Acceptance:** New allowances have unit tests at their caps; existing 40(1) results unchanged;
  a new `TAX_YEAR_*` can be added without touching `calcThaiTax`.

### 7. 🟢 English locale (i18n) — M
- **Why:** UI strings are hard-coded Thai; blocks non-Thai users and showcasing the project.
- **Where:** everywhere user-facing (screens + components). Introduce `src/i18n/` with a simple
  dictionary + `useT()` hook; start by extracting strings, not adding a heavy framework.
- **Approach:** Keep Thai as default; add `en`. Persist choice in `localStorage` next to theme
  (`src/state/useTheme.ts` is the pattern). Money/date formatting already centralised in
  `src/core/money.ts` and `src/db/data.ts`.
- **Acceptance:** Language switch in Settings; no hard-coded user-facing string left in the
  visited screens; both languages pass the a11y audit.

## Long-term / nice-to-have

- **8. 🟢 More report views** — net-worth-over-time line, cash-flow forecast, category trends.
  Build in `src/screens/Reports.tsx` with the existing hand-rolled SVG components
  (`src/components/Donut.tsx` is the model). Effort M.
- **9. 🟡 High-contrast / larger-text accessibility mode** — removes the one documented brand-accent
  AA exception for users who need it (`src/styles.css` tokens + a toggle). Effort S.
- **10. 🔴 Privacy-respecting usage counter** — e.g. GoatCounter/Plausible, opt-in, no PII.
  Needs an account + an explicit privacy decision (update `PRIVACY.md`). Effort S.
- **11. 🔴 Custom domain** — shorter, memorable URL. Owner decision + DNS. Effort S.
- **12. 🟡 Multi-currency** — only worthwhile with a clear use case; touches `money.ts`, every
  amount display, and the data model. Effort L. Currently a deliberate non-goal.

## Non-goals (kept out on purpose)

These were considered and declined to protect the local-first, minimal nature. Reopen only with
a strong reason:

- **OCR receipt scanning / bank API screen-scraping** — drags in a backend, cost, and data-privacy
  risk that contradicts the whole premise.
- **Double-entry accounting** — overkill for personal kakeibo; would bloat the model and UI.
- **Ads / tracking SDKs** — never. Privacy is the product.

## How to contribute a change

1. Read [`PROJECT_GUIDE.md`](PROJECT_GUIDE.md) for the file you're touching.
2. Put pure logic in `src/core/` with a colocated `.test.ts`; run `npm test`.
3. For schema changes, add a **new** `this.version(n)` in `src/db/db.ts` (never edit an applied
   one) and extend `src/core/backup.ts` (bump `schemaVersion`, keep import back-compatible).
4. Run `npm run build` (type-check) and `npm run test:e2e` (flows + axe a11y) before opening a PR.
5. CI (`.github/workflows/ci.yml`) runs all of the above; deploy is automatic on merge to `main`.
